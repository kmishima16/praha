# 基本的な設計原則

## はじめに

設計を考えられていないソフトウェアは技術的な負債が毎日蓄積して、開発速度が日に日に落ちていきます。開発のROIが悪化したサービスはユーザの要求に応えられなくなり徐々に競争力を失っていきます。

「設計を考える」と言われると複雑に感じるかもしれませんが、要は

- どこに何が書いてあるのか開発者同士がすぐに理解できること
- ソフトウェアの挙動に変更を加えなければいけない時に、変更箇所が最小限に留められる事」

この2点さえ実現できていれば、大半の問題が解決すると個人的には考えています。

そんな設計の基礎として重要な原理をいくつか理解しておけば、サービスを改修する時に変更箇所が1箇所、もしくは少なくて済むような、変更容易なソフトウェアに一歩近づけるはずです。ぜひこれを機に覚えておきましょう！
